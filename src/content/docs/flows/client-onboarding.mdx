<aside class="sb-tldr">
<p class="sb-tldr-label">TL;DR</p>
Four steps: <strong>create the client record (2 min) → add their contacts (2-3 min) → check the integration is connected (1 min, since most are workspace-level) → spin up their first brief from your house template (10-15 min)</strong>. Ship a test brief in the same sitting.
</aside>

This is the recipe for adding a new client once your workspace is set up and you already have a house template. For the very first client, follow [Getting Started](/getting-started/welcome/) instead.

## Before you start

Have these ready:

- **Client name** and **primary contact email**.
- **Client logo** if you white-label.
- **Access to integration data** *if you're using one*. In the current preview, the only working data integration is **ClickUp** (via Brief integrations). For monthly briefs that are mostly prose + tables, no integration is required.

## Step 1: Create the client record (2 minutes)

1. **Clients → New client**.
2. Fill in:
   - **Name** (display name in your nav).
   - **Website** (optional but useful for context).
   - **Logo** (optional; for white-label briefs).
   - **Primary contact name + email** (recommended — they'll be the default brief recipient).
3. Click **Create**.

You'll land on the client detail page.

## Step 2: Add the rest of the contacts (2-3 minutes)

From the client detail page, click **Contacts → Add contact** for each person:

| Role | Typical use | `receive reports` |
|---|---|---|
| **Primary** | The main day-to-day contact. | On |
| **Stakeholder** | Execs or specialists who want visibility. | Usually on |
| **Billing** | Handles invoices and money questions. | Usually off |
| **Technical** | Owns the data connection (e.g., the GA4 property admin). | Usually off |

Toggle **receive reports** for each based on whether that contact should be in the brief recipient list.

> **Reminder:** clients don't log in. Every contact reads briefs by email and via a signed-link portal. The role is for your organization; it doesn't change what they see.

## Step 3: Confirm data sources (1 minute)

Integrations are connected at the **workspace** level, not per-client. If your workspace already has Google connected (the typical case), you don't need to do anything here — the connection is already in place.

What changes per-client is which property/account you bind to in the brief editor. You'll do that in the next step.

If the new client uses data sources you haven't connected yet (e.g., a Slack channel, or one of the integrations on the roadmap), do that now from **Settings → Integrations**.

## Step 4: Build the first brief (10-15 minutes)

1. Click **Add to a brief** in the sidebar.
2. Pick the new client. SendBriefs resolves the current-period target:
   - If the client has a configured template + cadence (set via the **Default template** field on the client record), a draft brief is created for the period and you land in the editor.
   - Otherwise, you'll see the **deliberate authoring chooser** — pick **from template** to use your house template, or start a one-off.
3. Fill in the sections the template defines (the default starter has Done / In Progress / To Do / Additional Notes). Use the slash menu inside any section to add blocks.
4. Click **Preview** at the top of the editor to see how the brief reads. Spot-check branding and the period header.

## Step 5: Sanity-check before sending

There's no "send test to me" button on the brief itself in the current preview. To sanity-check rendering:

- **Use the Preview pane** for the in-app rendered version with branding.
- **Test the email wrapper** at Settings → Email Layout → open a layout → **Email myself** (this sends a sample with placeholder content).
- **Or use a "smoke-test client" pattern** — a client record with your own email on a `receive reports`-on contact. Send to that client first, then to the real one.

## Step 6: Send to the client

When the preview looks right:

1. Click **Send** on the brief.
2. Confirm the recipient list.
3. Hit Send.

The brief moves to `sent`. The engagement bar at the top fills in as deliveries land; the delivery card below shows per-recipient status.

> If your workspace requires internal approvals, the path is **Send → Submit for approval → approver clicks Approve → Send**. Adds a step but no more than a few minutes for a routine brief.

## Step 7: Move them into the recurring cycle

If recurring sends are available in your workspace and you have a steady cadence (most agencies do):

1. From the brief detail page, set up a **recurring schedule** matching your house cadence.
2. Done. Future periods will create + render + send automatically; you'll review them as part of your weekly cycle.

If recurring isn't available yet for your workspace, hold this client in your manual rotation — create + send each period from the template until the feature ships.

## A useful onboarding-day checklist

Walk through this when you onboard a new client:

- Client record created (name, website, logo)
- All contacts added with correct role + `receive reports` flag
- Integration available at the workspace level for the data the brief needs
- House template applied to the first brief
- Data blocks bound to the correct properties/accounts for this client
- Preview reviewed — numbers spot-checked against source platforms
- Test brief sent to your own email
- Test reviewed — looks right
- Real brief sent to client
- Recurring schedule set up (if available)
- Client added to your Friday review queue (mental note or actual calendar entry)

## Common bumps

### The client uses an integration that's not connected yet

If it's on our roadmap ([Integrations](/features/integrations/)) but not live, you can still build the brief with manual/prose content for that section until the integration ships. Or skip that brief block entirely for now.

### Your house template has blocks that don't apply to this client

Edit the brief (not the template) to remove or hide those blocks for this client. Don't fork the template just for one client unless you have multiple clients with the same pattern.

### Client doesn't have a logo yet, but you white-label

Leave the client logo blank — the workspace branding shows up instead. Add the logo later when you have it; previously-sent briefs are frozen so they keep the old branding, but new ones pick up the change.

### Client wants their own "from address"

The "from" address is set at the **workspace** level (Settings → Email Settings) and can optionally be overridden at the **template** level (so different report types can send from different addresses). There's no **per-client** sender address override today — if you need a per-client sending domain, the workarounds are:

- Create a dedicated template for that client with its own sender address.
- Or run that client in a separate workspace.

The **Custom domain** add-on is required for production custom-domain sending — talk to support to wire up the DNS.
