<aside class="sb-tldr">
<p class="sb-tldr-label">TL;DR</p>
Approvals can be required at the <strong>workspace</strong> level (default) or at the <strong>template</strong> level. Settings are <strong>off</strong>, <strong>on</strong>, or <strong>enforced_on</strong>. If either source says approval is required, it's required. Owner / Admin roles can approve by default — there's no per-client approver assignment in the current preview.
</aside>

## Where this lives

Approval configuration lives at **Settings → Approvals**. The page lets you:

- Set the workspace-wide default policy.
- Add per-client overrides (some clients require approval, others don't).
- See the resolved policy for any client at a glance.

Per-template settings live on each template (Settings → Templates → edit a template → Approval requirement).

## The three values

Approval requirement is a three-value setting, not a boolean:

### `off`
Approval is not required. The author can move a brief from `draft` → `sent` directly. The `pending_approval` state can still be entered manually if the author wants a review, but it's optional.

### `on`
Approval is required. Briefs must reach the `approved` state before they can be sent. This is the standard "we always want a second pair of eyes" setting.

### `enforced_on`
Approval is strictly enforced. Same as `on`, but with extra UI prevention so the author can't bypass approval (e.g., no "force send" affordance). Use when you have compliance or audit requirements.

## How sources combine

Approval comes from two sources, and the resolution rule is **OR**:

- **Workspace setting** (Settings → Approvals).
- **Template setting** (Settings → Templates → per-template approval requirement).

If **either** is `on` or `enforced_on`, approval is required. If both are `off`, approval is optional.

So you can have a workspace where most briefs send without approval (workspace = `off`) but specific high-stakes templates require it (template = `on`). Or the opposite — workspace defaults to required, with one fast-path template that opts out.

## Per-client overrides

The Approvals page lets you add **per-client overrides** for the workspace setting. Open the page, add a client to the override list, set the override. The override applies to all briefs for that client, regardless of which template they use.

This is the closest thing to per-client approver routing today. There's no UI to assign specific people as approvers for specific clients — the override controls whether approval is required, not who approves.

## Who can approve

Approval authority follows workspace roles:

| Role | Can approve? |
|---|---|
| **Owner** | Yes |
| **Admin** | Yes |
| **Member** | No (by default) |

There's no per-template approver assignment, no per-client approver assignment, and no granular `reports:approve` permission grant in the UI in the current preview. If you need a specific person to approve specific briefs, the closest workaround is convention plus the audit log.

## What submission and approval actually do

The state machine (also documented in [Edits & approvals](/flows/edits-and-approvals/)):

```
draft ───► pending_approval ───► approved ───► sent
              │      ▲
              ▼      │
         changes_requested
```

When an approver clicks **Approve**:

- Brief state → `approved`.
- An audit log entry records the approver, timestamp, and (optionally) a comment.
- A notification fires to the author (`report_approved`).
- The brief becomes sendable.

When an approver clicks **Request Changes**:

- Brief state → `changes_requested`.
- The approver enters a reason / note (free text, Tiptap-enabled).
- A notification fires to the author (`report_rejected`).
- The author can revise and resubmit (→ `pending_approval`) or discard and start over (→ `draft`).

## Approval comments and history

Every state transition is logged. The brief's history surfaces:

- Who submitted (and when).
- Who approved (and when, with optional comment).
- Who requested changes (and when, with reason).
- Who sent (and when, with recipient list).

For workspace-wide approval reporting (e.g., "show me every brief approved by Sam last quarter"), the audit log at `/activity?tab=advanced` has filters by action type, resource, user, and date range.

## Scheduled reports and approval

When a [scheduled report](/features/scheduled-reports/) fires:

- If the template requires approval (`on` or `enforced_on`), the brief is created in `draft` state and submitted for approval. It won't send until an approver acts.
- If the template doesn't require approval, the brief is created and auto-sent on the schedule.

This means a scheduled monthly brief on an `enforced_on` template effectively becomes: "create a draft on the schedule, then I review and approve it, then it sends." The schedule sets the cadence; approval still gates the send.

## Common gotchas

- **OR semantics surprise people.** If your workspace is `off` but a template is `on`, briefs from that template require approval. To fully disable approval, set both to `off`.
- **`enforced_on` vs `on` is subtle.** Both require approval before send. The distinction matters mostly for UI affordances and compliance posture. If you can't tell which one you need, `on` is usually right.
- **No per-client approver assignment.** If you need "Sam approves all Acme briefs," promote Sam to Admin and rely on convention + the audit log.
- **Approval doesn't auto-send.** Once approved, the brief still needs someone to click Send. For automated send-after-approval, use scheduled reports.
- **Changes requested doesn't preserve the brief's previous approval history through subsequent cycles.** Each new submission is a fresh approval cycle. If you need a longitudinal "this brief was rejected twice before being approved" record, check the audit log.
