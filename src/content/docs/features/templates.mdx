## What's in a template

A template is the reusable scaffold that new briefs are stamped from. Each template stores:

- **Identity** — name, description, slug, default/active flags.
- **Cadence** — `daily`, `weekly`, `biweekly`, `monthly`, `quarterly`, `milestone`, `adhoc`, or `custom`. The cadence drives period calculation and default title/slug formatting.
- **Period rules** — period start day, period length (days), and how the period date is auto-filled when a new brief is generated.
- **Approval requirement** — `off`, `on`, or `enforced_on` (see [Approval policy](/features/approvals/)).
- **Section definitions** — name, display name, email display name, color theme, icon, default content, placeholder text, and assigned contributor.
- **Email overrides** — sender name, sender address, signature, and email layout (the wrapper template that surrounds the brief in the email body).
- **Collaboration mode** — single-author vs. multi-contributor.
- **Contribution mode** — `append_no_context`, `append_with_context`, or `collaborative` (see [Add an update](/flows/add-an-update/)).
- **Section layout mode override** — `strict` or `flexible` (or unset to inherit from workspace).
- **Title/slug template variables** — placeholders like `{period.start}`, `{period.end}`, `{client.name}` that resolve when a new brief is created.

## Creating a template

Two paths:

### Save an existing brief as a template

The fastest way. Open any brief, use the menu → **Save as template**. The brief's block structure, sections, and section colors become the template. Set the cadence, name it, save. Subsequent briefs created from this template inherit the structure.

### Build a template from scratch

**Settings → Templates → Create Template**. Fill in:

1. **Template name** (max 200 chars) and **Description** (max 1000 chars).
2. **Cadence** — pick one of the 8 cadences.
3. **Sections** — open the Section Manager. The defaults are Done / In Progress / To Do / Additional Notes; add, remove, rename, recolor, and reorder as needed.
4. **Approval requirement** — leave `off` if briefs send without approval, `on` to require approval that can be overridden in some contexts, or `enforced_on` to strictly require it.
5. **Collaboration / contribution mode** — pick based on how your team works.
6. **Email overrides** — set per-template sender name, sender address, signature, or pick an email layout. Leave blank to inherit from workspace.
7. **Default / Active flags** — mark default if this is the template you want pre-selected for new briefs.

## Sections in detail

The Section Manager is where most of the editing happens. For each section:

| Field | Notes |
|---|---|
| **Name** | Internal identifier. Stable across renders. |
| **Display name** | Shown in the editor and (by default) in the rendered brief. |
| **Email display name** | Optional override for the section header in the email render. Useful when the in-app name is internal jargon. |
| **Color theme** | One of 17 hue families: red, orange, amber, yellow, lime, green, emerald, teal, cyan, sky, blue, indigo, violet, purple, fuchsia, pink, rose. |
| **Icon** | Emoji shown next to the section name. |
| **Placeholder text** | Shown when the section is empty in the editor — prompts the contributor. |
| **Default content** | Tiptap JSON that pre-fills the section when a new brief is generated. |
| **Default assignee** | The team member who's expected to fill this section in. Triggers an in-app notification when a new brief is created. |

Section CRUD is template-level. Changes apply to **future briefs** built from the template. Sent briefs stay frozen.

## Cadence semantics

| Cadence | What it means |
|---|---|
| `daily` | Daily standups / status. Auto-calculates "yesterday → today" or similar. |
| `weekly` | Week-ending summary. Configure the week start day (Sunday/Monday default). |
| `biweekly` | Every two weeks. |
| `monthly` | Calendar month. Period is the prior month. |
| `quarterly` | Calendar quarter (or fiscal — set the start month). |
| `milestone` | No fixed schedule; periods are project phases or deliverables. Period dates are author-controlled. |
| `adhoc` | One-off briefs. Creates `one_off` brief kind; no contribution window. |
| `custom` | User-defined start day + length. |

> **Cadence vs. scheduler.** The template cadence drives period math (what dates a new brief covers). The **scheduler** (Settings → Scheduling) is a separate concept that auto-creates briefs on a recurring cron — it supports `daily`, `weekly`, and `monthly` only. Templates with other cadences (biweekly / quarterly / milestone / adhoc / custom) require manual brief creation per period.

## Title / slug variables

Template titles and slugs support a handful of variables that resolve at brief creation:

- `{period.start}` — the period start date.
- `{period.end}` — the period end date.
- `{period.label}` — a human label for the period ("May 2026", "Q2 2026", "Week of May 18").
- `{client.name}` — the client name.

Use these in the **Title template** field so each new brief gets a meaningful title without manual entry.

## Cloning from a starter

The Templates page surfaces a small set of **starter templates** maintained by SendBriefs. Click a starter → **Clone** to copy it into your workspace. The clone has a "Starter" provenance badge so you can tell which templates came from the gallery vs. which you built. Edit freely after cloning — your changes don't affect the starter.

## Email-resolution chain

When a brief sends, the email's sender name, address, and signature resolve through this cascade (first non-null wins):

1. **Template-level override** (set on the template's email fields).
2. **Workspace default** (Settings → Email Settings).

The workspace can **lock** these settings (Settings → Email Settings → Lock email settings) — when locked, template overrides are ignored and workspace values always win.

The email **layout** (the Waypoint blocks that wrap the brief) is selected per template via the email-layout dropdown. Choose a layout you've built at Settings → Email Layout.

## Default template

One template can be marked as the **workspace default**. The default is what new briefs default to when no template is otherwise specified (e.g., when starting a brief from a client without a `default_template_id`).

Only one default at a time. Marking a new default doesn't automatically unmark the old one in some race conditions — if you see two templates flagged as default, edit the old one and uncheck the default flag.

## Active vs inactive

Templates can be set inactive without being deleted. Inactive templates:

- Don't appear in the brief-creation template picker.
- Don't appear in the scheduler's template dropdown.
- Existing briefs that were created from an inactive template are unaffected.

Use inactive when retiring a template you may want to revive later. Use delete when the template is truly dead.

## Common gotchas

- **Section changes don't retroactively edit sent briefs.** Each brief's section structure is frozen at brief-creation time. Editing the template affects future briefs only.
- **Email overrides don't apply if the workspace is locked.** If your template has a sender address override but the brief sends with the workspace address, check Settings → Email Settings → Lock email settings.
- **Title/slug variables that don't resolve render as literal text.** If you reference `{client.alias}` and the field doesn't exist, the title will literally read the raw `{client.alias}` text. Stick to documented variables.
- **Cloning a starter creates a snapshot.** Updates to the starter template don't propagate to your clone — that's by design. You own your edits.
- **Cadence is set once per template.** You can't switch a template from `monthly` to `quarterly` and keep all the old briefs in sync. Create a new template with the new cadence and move forward.
