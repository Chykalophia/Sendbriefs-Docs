A brief is a per-client, per-period report organized into named sections. The starter template ships with <strong>Done / In Progress / To Do / Additional Notes</strong>, but templates can add, rename, recolor, and reorder their own sections. The app sidebar labels these <strong>Reports</strong> — same thing.

## Anatomy of a brief

Every brief has three layers:

1. **Metadata** — client, reporting period, title, template (if any), recipients.
2. **Sections** — named groups (each rendered as a `section`-kind block). The starter sections are Done / In Progress / To Do / Additional Notes, but every template can customize them.
3. **Blocks inside each section** — text, headings, metrics, tables, callouts, images, etc. Stored as Tiptap JSON.

A brief is the rendered combination of these against the period at the moment it sends.

## Sections

Sections are defined by the brief's template. A template can:

- Set the section name and an **email display name** (used in the email render).
- Pick from **17 color themes** (red, orange, amber, yellow, lime, green, emerald, teal, cyan, sky, blue, indigo, violet, purple, fuchsia, pink, rose) and an icon (emoji).
- Provide **default content** so the section pre-fills when a new brief is generated.
- Define **placeholder text** to prompt contributors.
- Assign a **default contributor** so the right person gets notified to fill it in.

The **default starter sections** are:

| Section | Display | When to use |
|---|---|---|
| **Done** | ✅ Done | Work completed in this period. The hero section for most monthly briefs. |
| **In Progress** | 🔄 In Progress | Active work that hasn't shipped yet. |
| **To Do** | 📝 To Do | Planned work for the next period. |
| **Additional Notes** | 📌 Additional Notes | Context, callouts, blockers, asks — anything that doesn't fit the other three. |

Empty sections are hidden from the rendered output — you only render the sections you actually fill in.

To manage sections, open **Settings → Reports → Templates**, pick a template, and use the **Section manager** to add, remove, rename, reorder, and recolor them. Section changes are template-level: they apply to **future briefs** built from that template; existing sent briefs are frozen.

### Layout mode: strict vs flexible

Layout mode controls whether blocks can sit outside sections:

- **`strict`** — every block lives inside a section. Cleaner output, more structure. This is the default for new reports.
- **`flexible`** — top-level blocks (a hero callout, a divider) can sit alongside sections.

The resolved mode for any given brief comes from a four-layer cascade: workspace override → client override → template setting → brief's own value. Usually you can ignore this; the default works for most teams.

## The block editor

Briefs are built in a block editor. Inside any section, start typing for text, or hit **`/`** to open the slash-command menu.

<Placeholder type="screenshot" caption="Block editor — section header expanded, slash menu open showing block types" />

Block types available:

| Block | What it does |
|---|---|
| **Paragraph, Heading, Quote, List, Checklist** | Text and structure |
| **Metric, Gauge, Sparkline, StatsGrid** | Numeric callouts |
| **Table** | Tabular data |
| **Image** | Inline screenshots, branded assets |
| **Callout** | Highlight wins, blockers, asks |
| **Divider** | Visual section breaks |
| **Section** | Add a nested sub-section within a content bucket |
| **Column** | Side-by-side layout for two blocks |
| **Sentiment** | Status/sentiment indicator |
| **ChannelGrid** | Per-channel grid for multi-channel data |

Type `/` and start typing the block name to filter the picker.

> **About data-display blocks.** Metric / Gauge / Sparkline / StatsGrid are blocks for displaying numeric values. Today they're populated by you typing the values directly. The infrastructure for live data binding from external sources exists, but the only working data-pull integration in the current preview is ClickUp ambient ingest (and that auto-appends *content blocks*, not metric values). See [Integrations](/features/integrations/) for the full picture.

### Images and the Media Library

Image blocks reference URLs. To keep your assets organized:

- The **Media Library** at **Settings → Delivery → Media Library** is a workspace-wide upload + reference area. Upload once; reference the resulting URL from any brief's Image block.
- For one-off images, you can paste any externally-hosted URL into an Image block — but uploads through the Media Library are the durable path.

Media Library assets stay available even after a brief is sent. If you delete an asset that's referenced by a sent brief, the rendered brief in the recipient's inbox is unaffected (the email contains the raw HTML at send time), but the portal view may show a broken image.

## Brief state machine

```
draft ───► pending_approval ───► approved ───► sent
              │      ▲
              ▼      │
         changes_requested
```

Briefs can also move backwards (`sent → draft`, `approved → draft`, `pending_approval → draft`) — useful for fixing something and resending.

### `draft`
You can edit anything. Drafts auto-save as you type. No notifications fired.

### `pending_approval`
Author submitted the brief. Designated approvers (workspace policy decides — typically Owners and Admins) get a `report_submitted` notification.

### `changes_requested`
An approver requested changes. The brief goes back to the author with a note. Author fixes, resubmits — back to `pending_approval`.

### `approved`
Reviewer approved the brief. From here, anyone with send permission can move it to `sent`. Approval doesn't auto-send.

### `sent`
The brief was delivered. SendGrid handles email transport; we log the message ID and per-recipient delivery status (delivered, opened, clicked, bounced, portal-viewed).

> If your workspace doesn't require approvals, briefs go straight from `draft` to `sent` when you click Send.

## Creating and editing briefs

### Creating

The primary entry point is **Add to a brief** in the sidebar (route: `/reports/update`). This is a client-resolver flow that figures out the right draft for the client's current period — see [Send your first brief](/getting-started/send-first-brief/) for the walkthrough.

For one-off briefs or briefs from a specific template, when the resolver hits a non-Living client (no template / contributions disabled), it routes to a **deliberate authoring chooser** with "start a one-off" and "from template" options.

### Editing

Click any brief from the Reports list to open the editor.

Briefs in `draft` are fully editable. Briefs in `pending_approval`, `approved`, or `sent` are read-only by default — to edit, transition back to `draft` (the brief menu has a **Move to draft** option). The state transition is logged.

## Templates

Save any brief as a template. Templates remember:

- The section layout mode (strict / flexible).
- The block structure inside each of the four sections.
- The default **cadence** for any recurring brief built from this template: daily, weekly, biweekly, monthly, quarterly, milestone, ad-hoc, or custom.

Templates don't lock in the specific client or the data values. The next time you create a brief from this template, the structure is pre-filled.

Templates are managed at **Settings → Reports → Templates**.

## Recipients

A brief's recipient list is computed from the client's contacts where `receive reports` is on. You can review and tweak this list at send time, but the default is "every contact who's flagged to receive reports."

## Delivery and tracking

Once a brief is `sent`, two cards on the brief page show what happened:

- **Engagement bar** (top of the brief) — aggregate cadence, comment count, open count, open rate, and sent timestamp.
- **Delivery card** (below the brief) — per-recipient table with **Delivered** and **Opened** icons, plus aggregate Recipients / Delivered / Opened totals at the top.

The underlying SendGrid event trail (delivered, opened, clicked, bounced, spam-reported, unsubscribed) is captured for every recipient and surfaces in the delivery card's detail view.

## The hosted client portal

Every brief gets a signed portal URL embedded in the email. Recipients don't need to log in — the token in the URL grants access. Two flavors:

- **Single-brief**: `app.sendbriefs.com/portal/r/[token]`
- **Contact hub** (all briefs for a single contact): `app.sendbriefs.com/portal/c/[contact-token]`

Portal supports PDF download. For stricter access (verification code required), set the client's access tier to `verified` — see [Clients → portal access](/features/clients/#stricter-portal-access).

## Common gotchas

- **Section changes are template-level, not per-brief.** Editing a template's sections affects future briefs only; sent briefs are frozen at the moment they shipped. If you need a one-off variation, just edit blocks inside the brief.
- **Layout mode matters more than it sounds.** `strict` forces all content into sections. `flexible` allows top-level blocks. If a teammate's brief looks wildly different from yours, check the cascaded layout mode for that brief.
- **`changes_requested` keeps the brief editable.** The approver's note appears as an in-app notification + email. Edit and resubmit.
- **Recurring sends aren't on the brief.** They live in **Settings → Reports → Scheduling**, per client + frequency. The brief itself doesn't carry a schedule.
