## How integrations work today

There are two separate integration patterns in the product:

### A) Ambient ingest (the working one)

The **Brief integrations** card under **Settings → Integrations** lets you wire an external source to a brief template. When a new item appears in the source, it's auto-appended as a block in the matching draft brief's General section.

This is the integration system that's actually shipping. The setup is per-template: one mapping = one source selector → one brief template.

### B) OAuth connection storage

The **Integrations** card (also under Settings → Integrations) lets you complete an OAuth flow with Google or Slack. Credentials are stored encrypted; you'll see "Connected as: your.email@example.com."

This card exists, the OAuth flows complete, but **no other code in the product currently uses the stored credentials**. Connecting Google doesn't make GA4 data show up in your briefs. Connecting Slack doesn't enable notification routing through Slack. These are placeholder connections for features in development.

## Actually working today

### ClickUp (via Brief integrations)

**What it does:** Auto-pulls items from a ClickUp list and appends them as blocks in the matching brief template's draft. Items already ingested are tracked so re-runs don't duplicate.

**Setup:**

1. **Settings → Integrations → Brief integrations**.
2. Click **Add mapping**.
3. Enter:
   - Your ClickUp API token.
   - The ClickUp **list ID** to pull from.
   - The brief **template** that should receive the items.
4. Save. The mapping starts in `active` state.

**How it runs:**

- An ambient ingest job runs on a schedule (in production, via Upstash QStash). For each active mapping, it pulls new items and appends them to the most recent draft brief for that template.
- The job is idempotent — re-running on a set of items already ingested is a no-op.
- Mappings can be paused or deleted at any time.

**Status indicators:**

- **Active** — mapping is enabled and the job will pull on its next tick.
- **Paused** — mapping exists but the job skips it.
- **Error** — last run failed; check the error detail on the mapping row.

**Gotcha — schedule registration is a deploy step.** The ambient ingest job needs to be registered as a cron schedule (QStash or Vercel Cron) for it to actually run. If your workspace has mappings but nothing is ingesting, the schedule may not be wired up — let support know.

## Connection-only (no functional data flow yet)

These are visible in the **Integrations** card. The OAuth flows complete; credentials are stored. But the product doesn't currently use them for anything user-facing.

### Google

**What the marketing card says:** "Connect Google Analytics to track report engagement and metrics."

**What the OAuth scope requests:**

- `analytics.readonly`
- `userinfo.email`
- `userinfo.profile`

**What actually happens after you connect:** Credentials are stored against your workspace. No code path currently uses them — there is no GA4 data fetch in the product as of this preview. The integration is wired for a future Google-data-into-briefs feature.

If you connect Google, expect a **green "Connected" badge** and the email address of the connected account. Don't expect GA4 data to appear in briefs as a result.

### Slack

**What the marketing card says:** "Send report notifications and updates to Slack channels."

**What the OAuth scope requests:**

- `chat:write`
- `channels:read`
- `users:read`

**What actually happens after you connect:** Same as Google — credentials are stored, no other code path uses them. There is no Slack-send code in the product today.

The team's own code comment in the ingest service notes Slack adapter is on the to-do list ("*(Slack/Linear when their adapters land)*"). When the adapter ships, this page will be updated.

## Where this lives in the app

All integration management is under **Settings → Integrations**. Two cards live there:

- **Integrations** — the OAuth connect card (Google, Slack). Connect/disconnect, see status.
- **Brief integrations** — the per-template ambient-ingest mappings (currently ClickUp only). Wire a source selector to a brief template.

Visibility is gated by the `settings:update` permission — Owners and Admins see it; Members don't.

## Security and credentials

- **OAuth credentials** are encrypted at rest using AES-256-GCM with workspace-scoped keys.
- **ClickUp API tokens** are stored alongside the mapping, encrypted.
- **OAuth grants are audit-logged** — every `oauth.granted` event records who granted access, when, and which provider.
- **Disconnecting Google or Slack** revokes our stored token immediately.
- **Deleting a Brief integrations mapping** removes the stored API token from our system.

## Common gotchas

- **The Integrations card and the Brief integrations card are separate.** Connecting Google in the OAuth card doesn't create a Brief integrations mapping. You need ClickUp to set up an actual data flow today.
- **OAuth tokens expire.** Google's refresh-token flow is wired (the OAuth service has a `refreshAccessToken` method). Slack tokens currently have no refresh path — if revoked, reconnect.
- **The ambient ingest job needs a schedule.** If you set up a Brief integrations mapping and nothing ingests, ask support whether the QStash schedule is registered for your environment.
- **There's no per-client integration scope.** All integrations are workspace-level. A Brief integrations mapping wires to a *template*, not a client — that template can be used across multiple clients, but the source data comes from the one ClickUp list you specified.
