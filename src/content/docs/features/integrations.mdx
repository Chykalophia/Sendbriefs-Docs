Most of those aren't in the product code yet. This page reflects what's actually shipping, what's connection-only, and what's still roadmap.

## How integrations work today

There are two separate integration patterns in the product:

### A) Ambient ingest (the working one)

The **Brief integrations** card under **Settings → Integrations** lets you wire an external source to a brief template. When a new item appears in the source, it's auto-appended as a block in the matching draft brief's General section.

This is the integration system that's actually shipping. The setup is per-template: one mapping = one source selector → one brief template.

### B) OAuth connection storage

The **Integrations** card (also under Settings → Integrations) lets you complete an OAuth flow with Google or Slack. Credentials are stored encrypted; you'll see "Connected as: your.email@example.com."

This card exists, the OAuth flows complete, but **no other code in the product currently uses the stored credentials**. Connecting Google doesn't make GA4 data show up in your briefs. Connecting Slack doesn't enable notification routing through Slack. These are placeholder connections for features in development.

## Actually working today (with caveats)

### ClickUp ambient ingest (engine-only)

**What it does (in code):** A scheduled job pulls items from a ClickUp list and appends them as blocks in the matching brief template's current-period draft. Items already ingested are tracked so re-runs don't duplicate.

**What you can do as a user today:** Not much, directly. The **Brief integrations** card under **Settings → Account → Integrations** lets you create a *mapping* (template + integration + source selector). But the **integration** dropdown only lists already-connected OAuth providers — Google and Slack. **ClickUp isn't in the OAuth provider list**, so you can't self-serve a ClickUp credential through the UI.

In practice, ClickUp ambient ingest works for workspaces where someone with backend access has manually seeded a ClickUp integration row in the database. For everyone else, this is a "talk to support" path right now.

**How it runs once configured:**

- An ambient ingest job runs on a schedule (production: Upstash QStash). For each active mapping, it pulls new items and appends them to the most recent draft brief for that template.
- The job is idempotent — re-runs on already-ingested items are no-ops.
- Mappings can be paused or deleted from the Brief integrations card.

**Status indicators (on the mapping row):**

- **Active** — mapping is enabled; the job will pull on its next tick.
- **Paused** — mapping exists but the job skips it.
- **Error** — last run failed; check the detail on the row.

**Gotcha — schedule registration is a deploy step.** Even if a mapping exists, the cron schedule (QStash or Vercel Cron) has to be registered for the job to run. If your workspace has mappings but nothing is ingesting, the schedule may not be wired up for your environment — let support know.

## Connection-only (no functional data flow yet)

These are visible in the **Integrations** card. The OAuth flows complete; credentials are stored. But the product doesn't currently use them for anything user-facing.

### Google

**What the marketing card says:** "Connect Google Analytics to track report engagement and metrics."

**What the OAuth scope requests:**

- `analytics.readonly`
- `userinfo.email`
- `userinfo.profile`

**What actually happens after you connect:** Credentials are stored against your workspace. No code path currently uses them — there is no GA4 data fetch in the product as of this preview. The integration is wired for a future Google-data-into-briefs feature.

If you connect Google, expect a **green "Connected" badge** and the email address of the connected account. Don't expect GA4 data to appear in briefs as a result.

### Slack

**What the marketing card says:** "Send report notifications and updates to Slack channels."

**What the OAuth scope requests:**

- `chat:write`
- `channels:read`
- `users:read`

**What actually happens after you connect:** Same as Google — credentials are stored, no other code path uses them. There is no Slack-send code in the product today.

The team's own code comment in the ingest service notes Slack adapter is on the to-do list ("*(Slack/Linear when their adapters land)*"). When the adapter ships, this page will be updated.

## Where this lives in the app

All integration management is under **Settings → Account → Integrations**. Two cards live there:

- **Integrations** — the OAuth connect card. Available providers: **Google** and **Slack**. Connect, see status, disconnect.
- **Brief integrations** — per-template ambient-ingest mappings. Pick a brief template, pick a connected integration from the dropdown, enter a source selector. The mapping starts polling and appending new items into the matching draft brief.

Both cards are visible to Owners and Admins (`settings:update` permission).

## Security and credentials

- **OAuth credentials** are encrypted at rest using AES-256-GCM with workspace-scoped keys.
- **OAuth grants are audit-logged** — every `oauth.granted` event records who granted access, when, and which provider.
- **Disconnecting Google or Slack** revokes our stored token immediately.
- **Deleting a Brief integrations mapping** removes the mapping; the underlying integration row (the OAuth or API credential) stays connected.

## Common gotchas

- **The two cards are separate and don't talk to each other.** Connecting Google in the OAuth card doesn't do anything useful by itself — you'd still need a Brief integrations mapping AND a working data adapter for that provider.
- **The Brief integrations dropdown only shows Google and Slack today.** Neither provider has a working data adapter, so creating a mapping won't actually ingest anything until adapters land. ClickUp ambient ingest works in the back-end but isn't selectable from the UI.
- **OAuth tokens expire.** Google's refresh-token flow is wired. Slack tokens currently have no refresh path — if revoked, reconnect.
- **There's no per-client integration scope.** Integrations are workspace-level. Brief integrations mappings wire to a *template*, not a client — a template can be used across multiple clients, but each mapping has one source selector.
